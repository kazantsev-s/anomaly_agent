# Агент по поиску аномалий

## Предметная область

Данные были собраны с платформы [Kolesa.kz](https://kolesa.kz) - крупнейшей казахстанской онлайн-площадки автообъявлений по продаже авто. Основная доля объявлений - частные, чаще б/у. Количество объявлений достаточно большое, платформе важно, чтобы они были настоящими и корректно заполненными, чтобы пользователи в ленте видели как можно больше качественных объявлений, что повысило бы статус площадки, а продавцу важно как можно скорее найти покупателя.

В связи с этим агент будет искать среди опубликованных объявлений подозрительные, которые могут свидетельствовать о мошенничестве или некорректном заполнении данных.

## Реализация проекта

Проект представляет из себя AI-агента, который имеет доступ к данным, проводит полный анализ по поиску аномалий и умеет обрабатывать отдельные запросы пользователя. Для ответа на его вопросы и понимания представленных данных агент умеет формировать и выполнять SQL-запросы к базе данных, как заранее заготовленные, так и полностью кастомные, что является обязательным этапом при обработке запроса.

В проекте есть два режима работы:

- ответ на пользовательский вопрос через `/ask`;
- полный анализ аномалий таблицы через `/analyze`.

### Взаимодействие с агентом

Общение с агентом выстраивается через Telegram-бота, в котором работают следующие команды:

- `/start` - показывает краткое приветствие и подсказывает проверить подключение к базе.

- `/health` - проверяет доступность БД.

- `/ask {запрос}` - описание запроса для AI-агента (`/ask` и после пробела любой текстовый запрос).

- `/analyze` - запускает полный анализ таблицы по умолчанию: агент собирает схему и профиль данных, выполняет базовые проверки аномалий, при необходимости запускает дополнительные SQL-проверки и возвращает итоговый отчет.

- `/analyze {table_name}` - запускает полный анализ конкретной таблицы. Например, `/analyze kolesa_test_easy_001` позволяет проверить тестовый датасет с заранее внесенными аномалиями. Для тестовой таблицы после итогового отчета бот дополнительно считает качество поиска и отправляет отдельное сообщение с метриками.

### Используемый технологический стек

Проект написан на `Python`. В качестве СУБД используется `PostgreSQL`, запуск приложения и базы данных организован через `Docker` и `Docker Compose`.

Для работы с PostgreSQL используется асинхронная библиотека `asyncpg`: она отвечает за подключение к базе данных, инициализацию таблицы, загрузку данных из CSV и выполнение SQL-запросов.

Telegram-бот реализован с помощью библиотеки `aiogram`. Бот обрабатывает команды пользователя, проверяет подключение к базе данных и передаёт запросы AI-агенту.

Взаимодействие с LLM реализовано через официальный клиент `openai`, в частности через асинхронный клиент `AsyncOpenAI` и `Responses API`.

Логика AI-агента построена с использованием `LangGraph`. В проекте есть два графа:

- `ask_graph` - простой граф для команды `/ask`: генерация SQL-запроса, выполнение SQL и подготовка ответа пользователю.
- `analyze_graph` - граф полного анализа для команды `/analyze`: загрузка схемы, профилирование таблицы, сохранение прохода анализа, запуск проверок, дополнительные SQL-проверки, финальный отчет и оценка качества на тестовых датасетах.

Для загрузки переменных окружения используется `python-dotenv`.

### Логика работы графов

В проекте разделены две задачи: ответ на конкретный вопрос пользователя и полный анализ аномалий. Обе задачи собраны как графы `LangGraph`, где каждая нода отвечает за отдельный этап обработки.

`ask_graph` используется для команды `/ask`:

- `generate_sql_query` - получает вопрос пользователя и формирует SQL-запрос к таблице `kolesa`.
- `execute_sql` - выполняет сгенерированный SQL-запрос и сохраняет результат в состоянии графа.
- `answer` - передает исходный вопрос, SQL-запрос и результат выполнения агенту, чтобы получить итоговый ответ для пользователя.

Связь в графе линейная:

```text
START -> generate_sql_query -> execute_sql -> answer -> END
```

`analyze_graph` используется для команды `/analyze`:

- `load_schema` - берет имя таблицы из состояния графа, загружает фиксированную схему таблицы `kolesa` и раскладывает колонки по типам.
- `profile_table` - собирает профиль данных: количество строк, пропуски, уникальные значения, числовые статистики, частые текстовые значения и распределения boolean-полей. После этого профиль сохраняется в БД, а для анализа создается запись прохода.
- `run_standard_checks` - запускает базовые проверки: пропуски, `unknown`, пустые строки, числовые выбросы, невозможные годы, редкие или неожиданные категории, дубликаты и несогласованные поля.
- `plan_custom_checks` - получает компактный профиль, найденные проблемы и результаты уже выполненных custom SQL, после чего решает, нужны ли дополнительные SQL-проверки. Для каждого запроса агент указывает тип результата: поиск кандидатов в аномалии или контекст для объяснения найденных проблем.
- `run_custom_checks` - выполняет предложенные агентом read-only SQL-запросы и сохраняет их результаты или ошибки выполнения.
- `final_anomaly_answer` - сохраняет найденные findings, собирает стандартные findings, результаты custom SQL и формирует итоговый отчет по аномалиям. После успешного отчета проход закрывается со статусом `completed`, при ошибке - со статусом `failed`.
- `evaluate_analysis` - для тестовых таблиц сопоставляет ожидаемые аномалии с findings агента, сохраняет результат в таблицы оценки и готовит короткое сообщение с метриками.

После `plan_custom_checks` граф либо переходит к финальному отчету, либо запускает дополнительные SQL-проверки. Цикл `plan_custom_checks -> run_custom_checks -> plan_custom_checks` ограничен настройками из `.env`, чтобы анализ не выполнял бесконечное число запросов.

```text
START -> load_schema -> profile_table -> run_standard_checks -> plan_custom_checks
                                                            ├──> final_anomaly_answer -> evaluate_analysis -> END
                                                            └──> run_custom_checks -> plan_custom_checks
```

### Структура базы данных

Схема базы данных описана в `db.sql`. При старте приложения `init_db` создает таблицы, загружает основной CSV `data/kolesa.csv`, затем загружает тестовые CSV из `data/anomaly_cases/` и ожидаемые аномалии из `expected_anomalies_load.sql`.

Основная таблица:

- `kolesa` - базовая таблица объявлений. В ней хранятся поля объявления с Kolesa.kz: идентификатор, ссылка, дата парсинга, марка, модель, поколение, год, город, тип кузова, топливо, объем двигателя, пробег, коробка передач, привод, руль, цвет, регистрация, количество фото, цена и информация о найденном изображении.

Тестовые таблицы устроены так же, как `kolesa`, но содержат специально измененные данные:

- `kolesa_test_easy_001` - простые явные ошибки в значениях.
- `kolesa_test_logic_001` - логические конфликты между связанными полями.
- `kolesa_test_hard_001` - сложные контекстные аномалии, где значение может быть нормальным само по себе, но странным для конкретной модели, года или типа авто.
- `kolesa_test_mixed_001` - смешанный набор простых, логических и контекстных аномалий.

Таблицы для описания тестов:

- `test_datasets` - справочник тестовых датасетов: `test_id`, имя таблицы, сложность, пути к CSV и JSON с ожидаемыми аномалиями.
- `expected_anomalies` - список аномалий, которые были внесены в тестовые данные вручную. Здесь хранится тип аномалии, измененные колонки, старые и новые значения, селектор строки и описание ожидаемого обнаружения.

Таблицы для сохранения запусков анализа:

- `analysis_profiles` - снимок профиля таблицы на момент запуска анализа: количество строк, схема и рассчитанный профиль по колонкам.
- `analysis_runs` - история проходов агента. Запись связывается с `test_id`, если анализируется тестовая таблица, и с `profile_id`, чтобы потом можно было восстановить контекст запуска.
- `agent_findings` - найденные агентом аномалии. Для findings с конкретными примерами сохраняются `row_id` и `kolesa_id`, а агрегатная информация остается в `evidence` и `raw_finding`. Сюда попадают стандартные findings и успешные custom SQL-результаты типа `anomaly_candidates`, сохраненные как `custom_sql_anomaly`. Результаты типа `context` используются для объяснения аномалий и сохраняются в metadata прохода, но не создают отдельные findings.

Таблицы для последующей оценки качества:

- `evaluation_results` - итоговые метрики сравнения найденных аномалий с ожидаемыми: recall внутреннего поиска, recall итогового отчета, количество пропусков и findings вне тестовой разметки, которые не участвуют в recall.
- `anomaly_detection_matches` - связи между конкретными ожидаемыми аномалиями и findings агента, а также отметки о том, показана ли проблема в итоговом отчете.

### Тестовые датасеты

Тестовые датасеты лежат в `data/anomaly_cases/`. Для каждого набора есть CSV с измененными объявлениями и JSON с ожидаемыми аномалиями. SQL-файл `expected_anomalies_load.sql` загружает описания этих аномалий в таблицы `test_datasets` и `expected_anomalies`.

`easy_001` (`kolesa_test_easy_001`) содержит 7 явных аномалий, которые должны находиться стандартными проверками:

- невозможный будущий год выпуска;
- отрицательный пробег;
- нулевая цена;
- несуществующий тип топлива;
- пустая марка автомобиля;
- отрицательный объем двигателя;
- дублирующийся `kolesa_id`.

`logic_001` (`kolesa_test_logic_001`) содержит 5 логических конфликтов:

- `found_img = true`, но ссылка и имя файла изображения пустые;
- `found_img = false`, но ссылка на изображение заполнена;
- `imgs_count = 0`, но изображение считается найденным;
- бензиновый автомобиль с нулевым объемом двигателя;
- электрический автомобиль с ненулевым объемом двигателя.

`hard_001` (`kolesa_test_hard_001`) содержит 6 контекстных аномалий:

- Lexus LX 570 с объемом двигателя 1.5 л;
- Lexus LX 470 с кузовом хэтчбек;
- Hyundai Sonata 2026 года с пробегом 300000 км;
- BMW i3 как бензиновый автомобиль с объемом 2.0 л;
- Toyota Camry 2011 с ценой 45 млн тенге;
- Toyota Land Cruiser с передним приводом.

`mixed_001` (`kolesa_test_mixed_001`) содержит 9 аномалий из разных групп:

- невозможный будущий год;
- отрицательный пробег;
- несуществующий тип топлива;
- два конфликта по изображениям;
- Lexus LX 570 с маленьким объемом двигателя;
- Hyundai Sonata 2026 года с большим пробегом;
- BMW i3 как бензиновый автомобиль;
- Toyota Camry 2011 с сильно завышенной ценой.

Для проверки конкретного тестового набора можно передать имя таблицы в команду:

```text
/analyze kolesa_test_easy_001
```

При таком запуске проход сохранится в `analysis_runs`, профиль таблицы - в `analysis_profiles`, а найденные проблемы - в `agent_findings`. Если таблица есть в `test_datasets`, проход дополнительно будет связан с ее `test_id`, а после финального отчета запустится оценка качества.

```text
/ask Есть ли в данных подозрительные объявления в соотношении цены и года автомобиля. То есть условно новые автомобили по низкой цене
```

Был получен следующий ответ:

```text
Найдены подозрительные объявления по соотношению цены и года

Запрос искал автомобили:
• не старше примерно 5 лет, то есть с 2021 года
```

## Локальный запуск

Запуск происходит через Docker Compose. С отдельным поднятием БД и остальными сервисами.

### Подготовка

1. Скопировать `.env.example` в `.env`:

    *Mac/Linux:*

    ```bash
    cp .env.example .env
    ```

    *Windows:*

    ```bash
    copy .env.example .env
    ```

2. Вставить реальный `TG_TOKEN` в `.env`.

3. Вставьте OpenAI API ключ для взаимодействия с LLM от OpenAI (GPT)

4. Указать пароль для БД в `POSTGRES_PASSWORD`

Настройки полного анализа в `.env`:

- `ANALYZE_DEFAULT_TABLE` — таблица для команды `/analyze`.
- `ANALYZE_MAX_CUSTOM_CHECK_ITERATIONS` — максимум итераций дополнительных SQL-проверок.
- `ANALYZE_CUSTOM_SQL_PER_ITERATION_LIMIT` — максимум custom SQL-запросов за одну итерацию.
- `ANALYZE_CUSTOM_SQL_TOTAL_LIMIT` — общий лимит custom SQL-запросов за анализ.
- `ANALYZE_REPORT_FINDINGS_LIMIT` — максимум findings, передаваемых в итоговый отчет.
- `ANALYZE_REPORT_SAMPLE_ROWS_LIMIT` — максимум примеров строк для одного finding.

### Запуск контейнеров

Далее этапы запуска через Docker Compose контейнеров отдельно для базы данных PostgreSQL и остальных сервисов (агент, бот).

### Запуск PostgreSQL

1. Запуск контейнера PostgreSQL:

    ```bash
    docker compose -f docker-compose.db.yaml up -d
    ```

2. Посмотреть логи базы:

    ```bash
    docker compose -f docker-compose.db.yaml logs -f db
    ```

3. Проверить статус:

    ```bash
    docker compose -f docker-compose.db.yaml ps
    ```

4. Подключиться к базе данных:

    Используйте значения `.env`
    По умолчанию:

      - Host: `localhost`
      - Port: `5432`
      - Database: `kolesa`
      - User: `kolesa`
      - Password: `установленный_в_.env`

5. Остановить контейнер с БД

    Обычная остановка:

    ```bash
    docker compose -f docker-compose.db.yaml down
    ```

    Удалить базу вместе с данными:

    ```bash
    docker compose -f docker-compose.db.yaml down -v
    ```

### Запуск остальных сервисов

Может выполняться независимо от запуска БД

1. Запуск контейнера с выводом логов в терминал:

    Обычный запуск

    ```bash
    docker compose up --build
    ```

    Фоновый запуск:

    ```bash
    docker compose up --build -d
    ```

2. Посмотреть логи:

    ```bash
    docker compose logs -f
    ```

    Файловые логи сохраняются в `local/logs/logs.log`.

3. Проверить статус:

    ```bash
    docker compose ps
    ```

4. Остановить контейнер:

    ```bash
    docker compose down
    ```
